# Drive-Time Isochrone Workflow

This directory contains the script that generates a drive-time isochrone
(`60 minute` polygon) for every US ski resort in
`../research-data/us_ski_resorts.json`. The isochrones are computed against the
real road network via the OpenRouteService API, then rendered as a toggleable
overlay layer on `../map.html`.

## Why isochrones, not distance buffers

Naive Euclidean buffers ("draw a 200 km circle around each resort") badly
mis-estimate accessibility in the mountainous terrain on which ski resorts sit.
A 200 km straight-line buffer around Vail covers Aspen — but the actual drive
takes 4+ hours over Independence Pass. A 200 km straight-line buffer around
Mt. Bachelor covers Portland — but the actual drive is 3.5 hours over the
Cascades. Real isochrones, computed against the road network, are the standard
tool in spatial accessibility analysis.

## Why only 60 minutes

The OpenRouteService **free tier caps isochrone range at 3,600 seconds (60
minutes)**. Larger ranges (90 / 120 minutes, which would represent
weekend-trip rather than day-trip markets) require a paid ORS plan or
self-hosting OSRM/Valhalla against a US OSM extract.

The 60-minute polygon is the most analytically meaningful single band: it
represents the *local market* for each resort — the population that can
realistically reach it for a day trip without staying overnight. Empirically,
day-trip skier visits dominate total visit volume at most US resorts (the
exceptions are destination resorts in Colorado, Utah, Wyoming where
fly-in/multi-day visits dominate), so the 60-minute layer captures most of
the economically-relevant catchment for the long tail of community and
regional resorts.

## How to (re)generate the isochrones

1. **Get an OpenRouteService API key** (free, takes 2 minutes):

       https://openrouteservice.org/dev/#/signup

   Free tier: 2,500 requests/day, 40 req/min. Generating one full pass over
   ~217 resorts uses ~217 requests, well under the daily limit.

2. **Set the API key** and run the script. On Linux/Mac:

       export ORS_API_KEY='your-key-here'
       python3 compute_isochrones.py --ranges 3600

   On Windows PowerShell:

       $env:ORS_API_KEY = "your-key-here"
       python compute_isochrones.py --ranges 3600

   (Or pass the key directly: `python compute_isochrones.py --api-key YOUR_KEY --ranges 3600`)

   Wall clock time: ~6 minutes (rate-limited at ~40 req/min).
   Output file: `../research-data/resort_isochrones.geojson` (~1.4 MB).

3. **Resume after interruption.** The script saves checkpoints every 10
   resorts. If it crashes or you cancel:

       python3 compute_isochrones.py --ranges 3600 --resume

4. **Test with a small subset first** to confirm your API key works:

       python3 compute_isochrones.py --ranges 3600 --limit 5

## Output format

GeoJSON `FeatureCollection`. Each `Feature` is one resort's 60-minute
drive-time polygon. Properties:

```json
{
    "resort_name": "Vail",
    "state": "CO",
    "region": "Rocky Mountain",
    "value": 3600,
    "value_minutes": 60,
    "lat": 39.6403,
    "lon": -106.3742,
    "area_m2": 2527725859.81
}
```

## Known issues

A handful of resorts (~1% of the dataset) may fail to generate isochrones
because their location is poorly served by the OSM road network on which
ORS depends:

- **Eaglecrest** (Juneau, AK) — Juneau is the only US state capital with no
  road connection to the outside road network. ORS cannot compute drive-time
  polygons for road-isolated locations.

- **Teton Pass Ski Area** (Choteau, MT) — accessed via 10 miles of unpaved
  Forest Service road that may be incomplete in OSM.

These are data limitations, not script bugs. The map.html overlay simply
omits the failing resorts (their markers still appear; only the isochrone
polygon is missing).

## Reproducibility note

OpenRouteService rebuilds its road network periodically from OpenStreetMap.
Different runs of this script weeks apart may produce slightly different
isochrones (new roads, road closures, OSM edits). For a frozen reference
dataset, commit the generated `resort_isochrones.geojson` to git and treat
that as the canonical version.

## Cost / sustainability note

The free ORS tier is supported by HeiGIT (a non-profit at Heidelberg University)
and is intended for research and small-scale uses. If you regenerate isochrones
frequently or for much larger datasets, please consider hosting your own routing
engine (OSRM, Valhalla, or Graphhopper) on local OSM extracts, or contributing
back to HeiGIT.
